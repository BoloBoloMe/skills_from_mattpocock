# 编写 Skills

## 流程

1. **收集需求** - 询问用户：
   - 这个 skill 覆盖什么任务/领域？
   - 它应处理哪些具体用例？
   - 它需要可执行脚本，还是只需要说明？
   - 是否有要包含的参考材料？

2. **起草 skill** - 创建：
   - SKILL.md，包含简洁说明
   - 如果内容超过 500 行，创建额外参考文件
   - 如果需要确定性操作，创建实用脚本

3. **与用户审阅** - 展示草稿并询问：
   - 这是否覆盖了你的用例？
   - 是否有遗漏或不清楚的地方？
   - 是否有任何章节应更详细/更简略？

## Skill 结构

```
skill-name/
├── SKILL.md           # 主要说明（必需）
├── REFERENCE.md       # 详细文档（如需要）
├── EXAMPLES.md        # 使用示例（如需要）
└── scripts/           # 实用脚本（如需要）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力的简短描述。当出现[具体触发条件]时使用。
---

# Skill 名称

## 快速开始

[最小可运行示例]

## 工作流

[复杂任务的分步流程和检查清单]

## 高级功能

[链接到独立文件：见 [REFERENCE.md](REFERENCE.md)]
```

## `description` 要求

`description` 是智能体（agent）决定加载哪个 skill 时**唯一能看到的内容**。它会与所有其他已安装 skills 一起显示在系统提示（system prompt）中。你的智能体会读取这些 descriptions，并根据用户请求选择相关 skill。

**目标**：给你的智能体（agent）足够信息，使其知道：

1. 这个 skill 提供什么能力
2. 何时/为何触发它（具体关键词、上下文、文件类型）

**格式**：

- 最多 1024 个字符
- 使用第三人称编写
- 第一句：说明它做什么
- 第二句：说明“当 [具体触发条件] 时使用”

**好示例**：

```
从 PDF 文件提取文本和表格、填写表单、合并文档。当处理 PDF 文件，或用户提到 PDF、表单、文档提取时使用。
```

**坏示例**：

```
帮助处理文档。
```

坏示例无法让你的智能体（agent）区分它和其他文档 skills。

## 何时添加脚本

在以下情况添加实用脚本：

- 操作是确定性的（验证、格式化）
- 相同代码会被反复生成
- 需要显式处理错误

相比生成代码，脚本能节省 token 并提高可靠性。

## 何时拆分文件

在以下情况拆分为独立文件：

- SKILL.md 超过 100 行
- 内容包含不同领域（财务 vs 销售 schema）
- 高级功能很少需要

## 审阅清单

起草后，验证：

- [ ] `description` 包含触发条件（“当...时使用”）
- [ ] SKILL.md 少于 100 行
- [ ] 没有时效性信息
- [ ] 术语一致
- [ ] 包含具体示例
- [ ] 引用深度为一层
