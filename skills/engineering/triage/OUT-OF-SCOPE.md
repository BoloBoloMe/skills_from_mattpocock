# 范围外知识库

仓库中的 `.out-of-scope/` 目录用于存储被拒绝功能请求的持久记录。它有两个用途：

1. **组织记忆**——记录功能被拒绝的原因，这样关闭议题后理由不会丢失
2. **去重**——当新议题与先前拒绝的请求匹配时，此技能可以展示先前决策，而不是重新争论一遍

## 目录结构

```
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```

每个**概念**一个文件，而不是每个议题一个文件。请求同一事项的多个议题会归到同一个文件下。

## 文件格式

文件应采用宽松、可读的风格编写——更像一份简短设计文档，而不是数据库条目。使用段落、代码示例和例子，让首次接触它的人也能清楚理解其中的理由并觉得有用。

```markdown
# 深色模式

本项目不支持深色模式或面向用户的主题化。

## 为什么这不在范围内

渲染流水线假设使用 `ThemeConfig` 中定义的单一调色板。
支持多个主题将需要：

- 包裹整个组件树的主题上下文提供器
- 每个组件都支持感知主题的样式解析
- 用于用户主题偏好的持久化层

这是一项重大的架构变更，并不符合项目专注于内容创作的方向。
主题化是嵌入或重新分发输出的下游消费者需要关注的问题。

```ts
// 当前 ThemeConfig 接口并不是为运行时切换而设计的：
interface ThemeConfig {
  colors: ColorPalette; // 单一调色板，在构建时解析
  fonts: FontStack;
}
```

## 先前请求

- #42 — "添加深色模式支持"
- #87 — "用于无障碍的夜间主题"
- #134 — "深色主题选项"
```

### 文件命名

为概念使用简短、描述性的 短横线命名法（kebab-case）名称：`dark-mode.md`、`plugin-system.md`、`graphql-api.md`。名称应足够易识别，让浏览该目录的人无需打开文件也能理解被拒绝的是什么。

### 编写理由

理由应有实质内容——不是“我们不想要这个”，而是说明为什么。好的理由会引用：

- 项目范围或理念（“本项目专注于 X；主题化是下游关注点”）
- 技术约束（“支持此功能需要 Y，而 Y 与我们的 Z 架构冲突”）
- 战略决策（“我们选择使用 A 而不是 B，因为……”）

理由应具备持久性。避免引用临时情况（“我们现在太忙了”）——那些不是真正的拒绝，而是延期。

## 何时检查 `.out-of-scope/`

在分流（triage）期间（步骤 1：收集上下文），读取 `.out-of-scope/` 中的所有文件。评估新议题时：

- 检查该请求是否匹配已有的范围外概念
- 匹配依据是概念相似性，而不是关键词——“夜间主题”会匹配 `dark-mode.md`
- 如果匹配，请向维护者展示：“这与 `.out-of-scope/dark-mode.md` 类似——我们之前拒绝它的原因是 [理由]。你现在仍然持相同看法吗？”

维护者可以：

- **确认**——新议题会被添加到现有文件的“先前请求”列表，然后关闭
- **重新考虑**——范围外文件会被删除或更新，议题继续进入正常 triage
- **不同意**——这些议题相关但不同，继续正常分流（triage）

## 何时写入 `.out-of-scope/`

只有当一个 **增强（enhancement）**（不是 bug）被拒绝为 `wontfix` 时才写入。流程：

1. 维护者决定某个功能请求不在范围内
2. 检查是否已存在匹配的 `.out-of-scope/` 文件
3. 如果存在：将新议题追加到“先前请求”列表
4. 如果不存在：用概念名称、决策、理由和第一个先前请求创建新文件
5. 在议题上发布评论，解释该决策并提及 `.out-of-scope/` 文件
6. 使用 `wontfix` 标签关闭议题

## 更新或移除范围外文件

如果维护者改变了对先前被拒绝概念的看法：

- 删除 `.out-of-scope/` 文件
- 此技能不需要重新打开旧议题——它们是历史记录
- 触发重新考虑的新议题继续进入正常分流（triage）
