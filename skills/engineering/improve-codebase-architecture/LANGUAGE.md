# 语言

此技能提出的每条建议都使用共享词汇。精确使用这些术语——不要替换成“组件”、“服务”、“API”或“边界”。一致的语言就是重点。

## 术语

**模块**
任何具有接口和实现的东西。刻意不绑定尺度——同样适用于函数、类、包或跨层切片。
_避免_：单元、组件、服务。

**接口**
调用者为了正确使用模块而必须知道的一切。包括类型签名，也包括不变量、顺序约束、错误模式、所需配置和性能特征。
_避免_：API、签名（太窄——它们只指类型层面的表面）。

**实现**
模块内部的东西——它的代码主体。不同于**适配器**：一个东西可以是小适配器但有大实现（Postgres repo），也可以是大适配器但有小实现（内存 fake）。当话题是接缝时使用“适配器”；其他时候使用“实现”。

**深度**
接口处的杠杆效应——调用者（或测试）每学习一单位接口所能触发的行为量。当大量行为位于小接口之后时，模块是**深**的。当接口几乎和实现一样复杂时，模块是**浅**的。

**接缝**（来自 Michael Feathers）
一个可以改变行为而不必在该处编辑的地方。模块接口所在的*位置*。选择把接缝放在哪里，本身就是一个设计决策，不同于它后面放什么。
_避免_：边界（与 DDD 的 bounded context 过载）。

**适配器**
在接缝处满足接口的具体东西。描述的是*角色*（它填入哪个槽位），不是实体内容（里面是什么）。

**杠杆效应**
调用者从深度中获得的东西。他们每学习一单位接口，得到更多能力。一个实现会在 N 个调用点和 M 个测试中回本。

**局部性**
维护者从深度中获得的东西。变更、错误、知识和验证集中在一个地方，而不是散布到调用者中。修一次，处处修好。

## 原则

- **深度是接口的属性，不是实现的属性。** 深模块内部可以由小的、可 mock、可替换的部分组成——它们只是不属于接口。模块可以既有**内部接缝**（对其实现私有，由自己的测试使用），也有其接口处的**外部接缝**。
- **删除检验。** 想象删除该模块。如果复杂度消失了，这个模块没有隐藏任何东西（它只是透传）。如果复杂度在 N 个调用者中重新出现，这个模块就在发挥价值。
- **接口就是测试表面。** 调用者和测试跨过同一个接缝。如果你想测试越过接口之后的东西，该模块的形状可能是错的。
- **一个适配器意味着假想接缝。两个适配器意味着真实接缝。** 除非确实有东西会跨接缝变化，否则不要引入接缝。

## 关系

- 一个**模块**恰好有一个**接口**（它呈现给调用者和测试的表面）。
- **深度**是**模块**的属性，基于其**接口**衡量。
- **接缝**是**模块**的**接口**所在的位置。
- **适配器**位于**接缝**处，并满足**接口**。
- **深度**为调用者产生**杠杆效应**，为维护者产生**局部性**。

## 被拒绝的框架

- **把深度视为实现行数与接口行数之比**（Ousterhout）：这会奖励给实现填充代码。我们改用“深度即杠杆效应”。
- **把“接口”视为 TypeScript `interface` 关键字或类的 public 方法**：太窄——这里的接口包括调用者必须知道的每个事实。
- **“边界”**：与 DDD 的 bounded context 过载。说**接缝**或**接口**。
