# 诊断

一套处理疑难错误的纪律。只有在理由明确时才跳过阶段。

探索代码库时，使用项目的领域词汇表来建立相关模块的清晰心智模型，并检查你将触碰区域中的 ADR。

## 阶段 1 — 建立反馈循环

**这就是此技能的核心。** 其他一切都是机械执行。如果你拥有一个快速、确定、可由代理运行的通过/失败信号来捕捉该错误，你就会找到原因——二分、假设检验和加仪表都只是消费这个信号。如果没有，再怎么盯着代码看也救不了你。

在这里投入不成比例的精力。**要激进。要有创意。拒绝放弃。**

### 构造反馈循环的方法——大致按此顺序尝试

1. **失败测试**，放在任何能触达该错误的接缝处——单元、集成、端到端都可以。
2. 针对运行中的开发服务器编写 **Curl / HTTP 脚本**。
3. 使用固定输入进行 **CLI 调用**，将 stdout 与已知正确的快照做 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，并对 DOM/console/network 断言。
5. **重放捕获的跟踪。** 将真实网络请求 / payload / 事件日志保存到磁盘；在隔离环境中通过该代码路径重放。
6. **一次性测试支架。** 启动系统的最小子集（一个服务、mock 依赖），用单个函数调用触发错误代码路径。
7. **属性 / fuzz 循环。** 如果错误是“有时输出错误”，运行 1000 个随机输入并寻找失败模式。
8. **二分测试支架。** 如果错误出现在两个已知状态（commit、数据集、版本）之间，自动化“在状态 X 启动、检查、重复”，以便可以用 `git bisect run` 执行。
9. **差分循环。** 用同一输入运行旧版本与新版本（或两个配置），并对输出做 diff。
10. **HITL bash 脚本。** 最后的手段。如果必须有人点击，就用 `scripts/hitl-loop.template.sh` 驱动_他们_，让循环仍然结构化。捕获的输出会反馈给你。

建立正确的反馈循环，错误就已经修好 90%。

### 迭代反馈循环本身

把这个循环当成一个产品。一旦有了_一个_循环，就问：

- 我能让它更快吗？（缓存 setup、跳过无关初始化、缩小测试范围。）
- 我能让信号更尖锐吗？（针对具体症状断言，而不是“没有崩溃”。）
- 我能让它更确定吗？（固定时间、固定 RNG 种子、隔离文件系统、冻结网络。）

一个 30 秒且偶发失败的循环几乎不比没有循环好。一个 2 秒且确定的循环是调试超能力。

### 非确定性错误

目标不是干净复现，而是**更高的复现率**。将触发器循环 100 次、并行化、增加压力、缩窄时序窗口、注入 sleep。50% 偶发失败的错误可调试；1% 不可调试——持续提高复现率，直到它可调试。

### 当你确实无法建立循环时

停下来并明确说明。列出你尝试过的内容。向用户请求：(a) 能复现它的任何环境访问权限，(b) 捕获的工件（HAR 文件、日志 dump、core dump、带时间戳的屏幕录制），或 (c) 添加临时生产仪表的许可。没有循环时**不要**继续提出假设。

直到你拥有一个可信的循环之前，不要进入阶段 2。

## 阶段 2 — 复现

运行循环。观察错误出现。

确认：

- [ ] 循环产生的是**用户**描述的失败模式——不是恰好相邻的另一种失败。错误的 bug = 错误的修复。
- [ ] 该失败可在多次运行中复现（或者，对非确定性错误来说，以足够高的频率复现，足以用于调试）。
- [ ] 你已经捕获确切症状（错误消息、错误输出、缓慢耗时），以便后续阶段可以验证修复确实解决了它。

复现错误之前不要继续。

## 阶段 3 — 提出假设

在测试任何假设之前，生成 **3–5 个排序后的假设**。只生成一个假设会锚定在第一个看似合理的想法上。

每个假设都必须**可证伪**：说明它做出的预测。

> 格式：“如果 <X> 是原因，那么 <改变 Y> 会让错误消失 / <改变 Z> 会让错误更严重。”

如果你无法说明预测，这个假设只是感觉——丢弃它或把它打磨清楚。

**在测试前把排序列表展示给用户。** 他们通常拥有领域知识，能立刻重新排序（“我们刚刚部署了针对 #3 的变更”），或知道哪些假设已经被排除。这是低成本检查点，能节省大量时间。不要因此阻塞——如果用户暂时不在，就按你的排序继续。

## 阶段 4 — 加仪表

每个探针都必须映射到阶段 3 中的某个具体预测。**一次只改变一个变量。**

工具偏好：

1. 如果环境支持，优先使用**调试器 / REPL 检查**。一个断点胜过十条日志。
2. 在能区分假设的边界处添加**定向日志**。
3. 绝不要“记录一切再 grep”。

**为每条调试日志加标签**，使用唯一前缀，例如 `[DEBUG-a4f2]`。最后清理时只需一次 grep。未打标签的日志会存活；打标签的日志必须消亡。

**性能分支。** 对性能回退，日志通常是错误工具。改为：建立基线测量（计时支架、`performance.now()`、profiler、查询计划），然后二分。先测量，再修复。

## 阶段 5 — 修复 + 回归测试

在修复之前编写回归测试——但只有在存在**正确接缝**时才这样做。

正确接缝是指测试能按调用点处发生的方式触发**真实错误模式**。如果唯一可用的接缝太浅（错误需要多个调用者但测试只有单调用者、单元测试无法复现触发错误的调用链），那里的回归测试会带来虚假信心。

**如果不存在正确接缝，这本身就是发现。** 记录下来。代码库架构正在阻碍该错误被锁定。把这点标记给下一阶段。

如果存在正确接缝：

1. 将最小化复现转成该接缝处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 用原始（未最小化）场景重新运行阶段 1 的反馈循环。

## 阶段 6 — 清理 + 事后复盘

宣布完成前必须做到：

- [ ] 原始复现不再复现（重新运行阶段 1 的循环）
- [ ] 回归测试通过（或已记录不存在接缝）
- [ ] 所有 `[DEBUG-...]` 仪表都已移除（grep 该前缀）
- [ ] 一次性原型已删除（或移动到标记清楚的调试位置）
- [ ] 在 commit / PR 消息中说明最终被证明正确的假设——让下一个调试者能学习

**然后问：什么本可以防止这个错误？** 如果答案涉及架构变更（没有好的测试接缝、调用者纠缠、隐藏耦合），带着具体信息移交给 `/improve-codebase-architecture` 技能。建议要在修复完成**之后**提出，而不是之前——现在你拥有的信息比开始时更多。
