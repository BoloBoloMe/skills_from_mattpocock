# 深化

在给定依赖的情况下，如何安全地深化一组浅模块。假定使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**模块**、**接口**、**接缝**、**适配器**。

## 依赖类别

评估深化候选项时，对其依赖进行分类。类别决定了深化后的模块如何跨接缝测试。

### 1. 进程内

纯计算、内存状态、无 I/O。始终可以深化——合并这些模块，并直接通过新接口测试。不需要适配器。

### 2. 可本地替代

拥有本地测试替身的依赖（Postgres 的 PGLite、内存文件系统）。如果替身存在，就可以深化。深化后的模块在测试套件中与该替身一起运行测试。接缝是内部的；模块外部接口上没有端口。

### 3. 远程但自有（端口与适配器）

你自己的跨网络边界服务（微服务、内部 API）。在接缝处定义一个**端口**（接口）。深模块拥有逻辑；传输作为**适配器**注入。测试使用内存适配器。生产使用 HTTP/gRPC/queue 适配器。

推荐形态：*“在接缝处定义一个端口，为生产实现 HTTP 适配器，为测试实现内存适配器，这样即使它部署在网络两端，逻辑也位于一个深模块中。”*

### 4. 真正外部（mock）

你无法控制的第三方服务（Stripe、Twilio 等）。深化后的模块把外部依赖作为注入端口接收；测试提供 mock 适配器。

## 接缝纪律

- **一个适配器意味着假想接缝。两个适配器意味着真实接缝。** 除非至少有两个适配器是合理的（通常是生产 + 测试），否则不要引入端口。单适配器接缝只是间接层。
- **内部接缝与外部接缝。** 深模块可以既有内部接缝（对其实现私有，由自己的测试使用），也有位于其接口处的外部接缝。不要仅仅因为测试使用内部接缝，就把它们暴露到接口中。

## 测试策略：替换，不叠加

- 一旦存在针对深化模块接口的测试，浅模块上的旧单元测试就变成浪费——删除它们。
- 在深化模块的接口处编写新测试。**接口就是测试表面**。
- 测试通过接口断言可观察结果，而不是内部状态。
- 测试应能经受内部重构——它们描述行为，而不是实现。如果实现改变时测试也必须改变，那它测试越过了接口。
