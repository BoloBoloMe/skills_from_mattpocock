# 设计两次

当用户想为一个已选择的深化候选对象探索替代接口时, 使用这种并行子代理模式. 基于 "Design It Twice" (Ousterhout) — 你的第一个想法不太可能是最好的.

使用 [SKILL.md](SKILL.md) 中的词汇 — **模块**, **接口**, **接缝**, **适配器**, **杠杆效应**.

## 流程

### 1. 界定问题空间

在生成子代理之前, 为所选择的候选对象写一段面向用户的问题空间说明:

* 任何新接口都需要满足的约束
* 它会依赖的依赖项, 以及它们分别属于哪个类别 (见 [DEEPENING.md](DEEPENING.md))
* 一个粗略的说明性代码草图, 用来支撑这些约束 — 不是提案, 只是让约束变得具体的一种方式

向用户展示这部分内容, 然后立即进入第 2 步. 用户在阅读和思考的同时, 子代理并行工作.

### 2. 生成子代理

使用 Agent 工具并行生成 3+ 个子代理. 每个子代理都必须为深化后的模块产出一个 **根本不同** 的接口.

为每个子代理提供一份单独的技术简报 (文件路径, 耦合细节, 来自 [DEEPENING.md](DEEPENING.md) 的依赖类别, 接缝背后是什么). 该简报独立于第 1 步中面向用户的问题空间说明. 给每个代理一个不同的设计约束:

* Agent 1: "最小化接口 — 目标最多 1–3 个入口点. 最大化每个入口点的杠杆效应."
* Agent 2: "最大化灵活性 — 支持许多用例和扩展."
* Agent 3: "为最常见的调用方优化 — 让默认场景变得微不足道."
* Agent 4 (如果适用): "围绕跨接缝依赖的端口 & 适配器进行设计."

在简报中同时包含 [SKILL.md](SKILL.md) 词汇和 CONTEXT.md 词汇, 这样每个子代理都会用与架构语言和项目领域语言一致的方式命名事物.

每个子代理输出:

1. 接口 (类型, 方法, 参数 — 加上不变量, 顺序, 错误模式)
2. 展示调用方如何使用它的使用示例
3. 实现隐藏在接缝背后的内容
4. 依赖策略和适配器 (见 [DEEPENING.md](DEEPENING.md))
5. 权衡 — 哪里杠杆效应高, 哪里薄弱

### 3. 展示并比较

按顺序展示各个设计, 这样用户可以消化每一个设计, 然后用散文形式比较它们. 按 **深度** (接口处的杠杆效应), **本地性** (变化集中在哪里), 以及 **接缝位置** 进行对比.

比较之后, 给出你自己的建议: 你认为哪个设计最强, 以及原因. 如果不同设计中的元素可以很好地组合, 提出一个混合方案. 要有主见 — 用户想要的是强判断, 而不是菜单.
