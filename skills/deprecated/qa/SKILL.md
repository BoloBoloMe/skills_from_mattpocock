# QA 会话

运行交互式 QA 会话。用户描述他们遇到的问题。你负责澄清、探索代码库以获取上下文，并创建持久、以用户为中心且使用项目领域语言的 GitHub issue。

## 针对用户提出的每个问题

### 1. 倾听并轻量澄清

让用户用自己的话描述问题。**最多问 2–3 个简短澄清问题**，重点关注：

- 他们期望发生什么 vs 实际发生了什么
- 复现步骤（如果不明显）
- 问题是稳定出现还是偶发

不要过度盘问。如果描述已经足够创建 issue，就继续下一步。

### 2. 在后台探索代码库

与用户交谈时，在后台启动一个 Agent（subagent_type=Explore）以理解相关区域。目标不是寻找修复方案，而是：

- 学习该区域使用的领域语言（检查 UBIQUITOUS_LANGUAGE.md）
- 理解该功能本应做什么
- 识别面向用户的行为边界

这些上下文能帮助你写出更好的 issue，但 issue 本身不应引用具体文件、行号或内部实现细节。

### 3. 评估范围：单个 issue 还是拆分？

创建前，判断这是**单个 issue**，还是需要**拆分**为多个 issue。

以下情况需要拆分：

- 修复跨越多个独立区域（例如“表单校验错误，并且成功消息缺失，并且重定向坏了”）
- 存在明显可分离的关注点，不同人员可以并行处理
- 用户描述的内容有多个不同的失败模式或症状

以下情况保持为单个 issue：

- 这是某个位置的一项错误行为
- 所有症状都由同一个根本行为导致

### 4. 创建 GitHub issue

使用 `gh issue create` 创建 issue。不要先让用户审阅——直接创建并分享 URL。

Issue 必须**持久**——即使经历重大重构后仍应有意义。从用户视角来写。

#### 单个 issue

使用此模板：

```
## 发生了什么

[用通俗语言描述用户实际经历的行为]

## 我期望什么

[描述期望行为]

## 复现步骤

1. [开发者可以遵循的具体编号步骤]
2. [使用代码库中的领域术语，而不是内部模块名]
3. [包含相关输入、标志或配置]

## 附加上下文

[来自用户或代码库探索的任何额外观察，用于帮助界定 issue——例如“这只在使用 Docker layer 时发生，filesystem layer 不会”——使用领域语言，但不要引用文件]
```

#### 拆分（多个 issue）

按依赖顺序创建 issue（阻塞项优先），这样你可以引用真实的 issue 编号。

每个子 issue 使用此模板：

```
## 父 issue

#<parent-issue-number>（如果你创建了跟踪 issue）或“QA 会话中报告”

## 哪里不对

[描述这个具体的行为问题——只描述这一片，不要描述整个报告]

## 我期望什么

[针对这个具体片段的期望行为]

## 复现步骤

1. [仅针对此 issue 的步骤]

## 被阻塞于

- #<issue-number>（如果此 issue 必须等另一个解决后才能修复）

如果没有阻塞项，则写“无——可以立即开始”。

## 附加上下文

[与此片段相关的任何额外观察]
```

创建拆分时：

- **宁可多建薄 issue，也不要少建厚 issue**——每个都应可独立修复和验证
- **诚实标记阻塞关系**——如果 issue B 确实必须等 issue A 修好后才能测试，就说明。如果它们独立，则都标为“无——可以立即开始”
- **按依赖顺序创建 issue**，这样可以在“被阻塞于”中引用真实 issue 编号
- **最大化并行度**——目标是让多个人（或智能体）可以同时领取不同 issue

#### 所有 issue 正文的规则

- **不要包含文件路径或行号**——这些会过时
- **使用项目的领域语言**（如果存在，检查 UBIQUITOUS_LANGUAGE.md）
- **描述行为，而不是代码**——写“sync service 无法应用 patch”，不要写“applyPatch() 在第 42 行抛错”
- **复现步骤是必需的**——如果无法确定，就询问用户
- **保持简洁**——开发者应能在 30 秒内读完 issue

创建后，打印所有 issue URL（并总结阻塞关系），然后询问：“下一个问题，还是结束？”

### 5. 继续会话

持续进行，直到用户说完成。每个 issue 都是独立的——不要批量处理。
