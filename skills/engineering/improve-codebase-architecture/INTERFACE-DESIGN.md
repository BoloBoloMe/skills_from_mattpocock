# 接口设计

当用户想为选定的深化候选项探索替代接口时，使用这种并行子代理模式。基于 “Design It Twice”（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**模块**、**接口**、**接缝**、**适配器**、**杠杆效应**。

## 流程

### 1. 框定问题空间

在启动子代理之前，为选定候选项写一段面向用户的问题空间说明：

- 任何新接口都需要满足的约束
- 它会依赖的依赖项，以及它们分别属于哪个类别（见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的说明性代码草图，用来锚定约束——不是提案，只是让约束具体化的方式

把这展示给用户，然后立即进入步骤 2。用户阅读和思考的同时，子代理并行工作。

### 2. 启动子代理

使用 Agent 工具并行启动 3 个以上子代理。每个子代理都必须为深化后的模块产出一个**根本不同**的接口。

用单独的技术简报提示每个子代理（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝后面是什么）。该简报独立于步骤 1 中面向用户的问题空间说明。给每个代理不同的设计约束：

- 代理 1：“最小化接口——目标最多 1–3 个入口点。最大化每个入口点的杠杆效应。”
- 代理 2：“最大化灵活性——支持许多用例和扩展。”
- 代理 3：“为最常见调用者优化——让默认情况微不足道。”
- 代理 4（如适用）：“围绕端口与适配器设计跨接缝依赖。”

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，让每个子代理在命名时同时符合架构语言和项目领域语言。

每个子代理输出：

1. 接口（类型、方法、参数——加上不变量、顺序、错误模式）
2. 展示调用者如何使用它的用法示例
3. 实现在接缝后隐藏了什么
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——哪里杠杆效应高，哪里薄弱

### 3. 展示并比较

按顺序展示设计，让用户能够吸收每一个，然后用文字比较它们。按**深度**（接口处的杠杆效应）、**局部性**（变更集中在哪里）和**接缝放置**进行对比。

比较之后，给出你自己的建议：你认为哪个设计最强，以及为什么。如果不同设计中的元素适合组合，就提出一个混合方案。要有立场——用户想要的是有力判断，而不是菜单。
