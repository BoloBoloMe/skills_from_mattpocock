# 通用语言

从当前对话中提取并形式化领域术语，形成一致的词汇表，并保存到本地文件。

## 流程

1. **扫描对话**，寻找与领域相关的名词、动词和概念
2. **识别问题**：
   - 同一个词用于不同概念（歧义）
   - 不同词用于同一概念（同义词）
   - 含糊或过载的术语
3. **提出规范词汇表**，并做出有主见的术语选择
4. **写入工作目录中的 `UBIQUITOUS_LANGUAGE.md`**，使用下面的格式
5. **在对话中内联输出摘要**

## 输出格式

写入一个具有以下结构的 `UBIQUITOUS_LANGUAGE.md` 文件：

```md
# 通用语言

## 订单生命周期

| 术语          | 定义                             | 应避免的别名     |
| ------------- | -------------------------------- | ---------------- |
| **订单**      | 客户购买一个或多个商品的请求     | 购买、交易       |
| **发票**      | 交付后发送给客户的付款请求       | 账单、付款请求   |

## 人员

| 术语          | 定义                             | 应避免的别名       |
| ------------- | -------------------------------- | ------------------ |
| **客户**      | 下订单的个人或组织               | 客户端、买家、账户 |
| **用户**      | 系统中的认证身份                 | 登录、账户         |

## 关系

- 一张**发票**只属于一个**客户**
- 一个**订单**会产生一张或多张**发票**

## 示例对话

> **开发者：** “当**客户**下**订单**时，我们会立即创建**发票**吗？”
> **领域专家：** “不会——只有在确认**履约**后才会生成**发票**。如果商品分多个**发货批次**发出，一个**订单**可以产生多张**发票**。”
> **开发者：** “所以如果一个**发货批次**在发出前被取消，就不会为它创建**发票**？”
> **领域专家：** “没错。**发票**生命周期绑定的是**履约**，不是**订单**。”

## 已标记的歧义

- “account” 曾同时表示**客户**和**用户**——它们是不同概念：**客户**下订单，而**用户**是系统中的认证身份，可能代表也可能不代表一个**客户**。
```

## 规则

- **要有主见。** 当多个词表示同一概念时，选择最佳词，并把其他词列为应避免的别名。
- **明确标记冲突。** 如果某个术语在对话中被含混使用，在“已标记的歧义”部分指出，并给出清晰建议。
- **只包含领域专家关心的术语。** 跳过模块或类的名称，除非它们在领域语言中有含义。
- **定义要紧凑。** 最多一句话。定义它是什么，而不是它做什么。
- **展示关系。** 使用粗体术语名，并在明显时表达基数关系。
- **只包含领域术语。** 跳过通用编程概念（array、function、endpoint），除非它们有领域特定含义。
- **在自然聚类出现时分成多个表格**（例如按子域、生命周期或参与者）。每个分组都有自己的标题和表格。如果所有术语都属于单一内聚领域，一个表格即可——不要强行分组。
- **写一个示例对话。** 写一段开发者与领域专家之间的短对话（3–5 轮），展示术语如何自然互动。对话应澄清相关概念之间的边界，并展示术语的精确用法。

<example>

## 示例对话

> **开发者：** “我如何在没有 Docker 的情况下测试 **sync service**？”

> **领域专家：** “提供 **filesystem layer**，而不是 **Docker layer**。它实现相同的 **Sandbox service** 接口，但使用本地目录作为 **sandbox**。”

> **开发者：** “所以 **sync-in** 仍然会创建一个 **bundle** 并解包它？”

> **领域专家：** “没错。**sync service** 不知道自己在和哪个 layer 通信。它调用 `exec` 和 `copyIn`——**filesystem layer** 只是把这些作为本地 shell 命令运行。”

</example>

## 重新运行

在同一对话中再次调用时：

1. 读取现有的 `UBIQUITOUS_LANGUAGE.md`
2. 纳入后续讨论中的任何新术语
3. 如果理解发生演进，更新定义
4. 重新标记任何新的歧义
5. 重写示例对话，以纳入新术语
