# 编写智能体简报（Agent Brief）

智能体简报（agent brief）是当 GitHub 议题移动到 `ready-for-agent` 时发布在该议题上的结构化评论。它是 AFK agent 将依据的权威规格说明。原始议题正文和讨论是上下文——智能体简报才是契约。

## 原则

### 持久性优先于精确性

议题可能会在 `ready-for-agent` 中停留数天或数周。在此期间代码库会变化。编写简报（brief）时要让它即使在文件被重命名、移动或重构后仍然有用。

- **要**描述接口、类型和行为契约
- **要**指出智能体应查找或修改的具体类型、函数签名或配置形状
- **不要**引用文件路径——它们会过时
- **不要**引用行号
- **不要**假设当前实现结构会保持不变

### 描述行为，而非步骤

描述系统应该做**什么**，而不是**如何**实现它。智能体会重新探索代码库并自行做出实现决策。

- **好：** “`SkillConfig` 类型应接受一个类型为 `CronExpression` 的可选 `schedule` 字段”
- **坏：** “打开 src/types/skill.ts，并在第 42 行添加 schedule 字段”
- **好：** “当用户不带参数运行 `/triage` 时，他们应看到需要关注的议题摘要”
- **坏：** “在主处理函数中添加 switch 语句”

### 完整的验收标准

智能体需要知道何时算完成。每份智能体简报（agent brief）都必须有具体、可测试的验收标准。每条标准都应可独立验证。

- **好：** “运行 `gh issue list --label needs-triage` 会返回已经过初始分类的议题”
- **坏：** “Triage 应正常工作”

### 明确的范围边界

说明哪些内容不在范围内。这可以防止智能体镀金或对相邻功能做出假设。

## 模板

```markdown
## Agent Brief

**类别：** bug / enhancement
**摘要：** 对需要发生的事情做一句话描述

**当前行为：**
描述现在会发生什么。对于 bug，这是损坏的行为。
对于 enhancement，这是该功能所基于的现状。

**期望行为：**
描述 agent 的工作完成后应该发生什么。
请具体说明边界情况和错误条件。

**关键接口：**
- `TypeName`——需要改变什么以及为什么
- `functionName()` 返回类型——当前返回什么，以及应该返回什么
- 配置形状——所需的任何新配置选项

**验收标准：**
- [ ] 具体、可测试的标准 1
- [ ] 具体、可测试的标准 2
- [ ] 具体、可测试的标准 3

**范围外：**
- 本议题中不应更改或处理的事项
- 看起来相关但实际独立的相邻功能
```

## 示例

### 好的智能体简报（agent brief，bug）

```markdown
## Agent Brief

**类别：** bug
**摘要：** 技能描述截断会截断到单词中间，产生破损输出

**当前行为：**
当技能描述超过 1024 个字符时，无论单词边界如何，都会在正好
1024 个字符处截断。这会产生以半个单词结尾的描述
（例如“当用户想要配”）。

**期望行为：**
截断应在 1024 个字符前的最后一个单词边界处断开，
并追加 "..." 以表示已截断。

**关键接口：**
- `SkillMetadata` 类型的 `description` 字段——无需类型变更，
  但填充它的验证/处理逻辑需要遵守单词边界
- 任何读取 SKILL.md frontmatter 并提取 description 的函数

**验收标准：**
- [ ] 低于 1024 个字符的描述保持不变
- [ ] 超过 1024 个字符的描述在 1024 个字符前的最后一个单词边界处截断
- [ ] 截断后的描述以 "..." 结尾
- [ ] 包含 "..." 在内的总长度不超过 1024 个字符

**范围外：**
- 更改 1024 字符限制本身
- 支持多行 description
```

### 好的智能体简报（agent brief，enhancement）

```markdown
## Agent Brief

**类别：** enhancement
**摘要：** 添加 `.out-of-scope/` 目录支持，用于跟踪被拒绝的功能请求

**当前行为：**
当功能请求被拒绝时，议题会带着 `wontfix` 标签和一条评论被关闭。
没有对该决策或理由的持久记录。未来类似请求需要维护者回忆或搜索先前讨论。

**期望行为：**
被拒绝的功能请求应记录在 `.out-of-scope/<concept>.md` 文件中，
这些文件捕获决策、理由，以及请求该功能的所有议题链接。
在 triage 新议题时，应检查这些文件以寻找匹配项。

**关键接口：**
- `.out-of-scope/` 中的 Markdown 文件格式——每个文件应有一个
  `# Concept Name` 标题、一行 `**Decision:**`、一行 `**Reason:**`，
  以及带议题链接的 `**Prior requests:**` 列表
- triage 工作流应尽早读取所有 `.out-of-scope/*.md` 文件，
  并按概念相似性将传入议题与它们匹配

**验收标准：**
- [ ] 将功能作为 wontfix 关闭时，会在 `.out-of-scope/` 中创建/更新文件
- [ ] 文件包含决策、理由和已关闭议题的链接
- [ ] 如果已存在匹配的 `.out-of-scope/` 文件，新议题会追加到其“先前请求”列表，而不是创建重复文件
- [ ] 在 triage 期间，会检查现有 `.out-of-scope/` 文件，并在新议题匹配先前拒绝时展示出来

**范围外：**
- 自动匹配（由人确认匹配）
- 重新打开先前被拒绝的功能
- Bug 报告（只有 enhancement 拒绝会进入 `.out-of-scope/`）
```

### 差的智能体简报（agent brief）

```markdown
## Agent Brief

**摘要：** 修复 triage bug

**要做什么：**
triage 那部分坏了。看看主文件并修复它。
第 150 行附近的函数有问题。

**要修改的文件：**
- src/triage/handler.ts（第 150 行）
- src/types.ts（第 42 行）
```

这很差，因为：
- 没有类别
- 描述含糊（“triage 那部分坏了”）
- 引用了会过时的文件路径和行号
- 没有验收标准
- 没有范围边界
- 没有描述当前行为与期望行为
