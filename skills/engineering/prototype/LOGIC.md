# 逻辑原型

一个微型交互终端应用, 让用户手动驱动状态模型. 当问题是关于 **业务逻辑, 状态转换, 或数据形状** 时使用 - 那种在纸上看起来合理, 但只有当你用真实案例推过它才会感觉不对的东西.

## 这是正确形状的情况

- "我不确定这个状态机是否处理了 X 然后 Y 的边缘情况."
- "这个数据模型是否真的让我表示那种情况..."
- "我想在写它之前感受一下 API 应该是什么样."
- 任何用户想要 **按按钮并观察状态变化** 的情况.

如果问题是 "这应该看起来怎样" - 错误分支. 使用 [UI.md](UI.md).

## 流程

### 1. 陈述问题

在写代码之前, 写下你正在原型化什么状态模型和什么问题. 一段话, 在原型的 README 或文件顶部注释中. 一个回答错误问题的 logic prototype 是纯粹浪费 - 让问题明确, 以便后续可以检查, 无论用户现在在看还是 AFK 后返回.

### 2. 选择语言

使用宿主项目使用的任何语言. 如果项目没有明显运行时 (例如 docs repo), 询问.

匹配项目已有的工具约定 - 不要仅仅为原型添加新的包管理器或运行时.

### 3. 将逻辑隔离在一个可移植模块中

把实际逻辑 - 回答问题的那一部分 - 放在一个小的, 纯 interface 背后, 以后可以提升并放入真实代码库. 围绕它的 TUI 是一次性的; 逻辑模块不应该这样.

正确的形状取决于问题:

- **一个纯 reducer** - `(state, action) => state`. 适合当动作是离散事件且状态是单一值的情况.
- **一个状态机** - 显式状态和转换. 适合当 "现在哪些动作是合法的" 是问题的一部分的情况.
- **一组小的纯函数** 对一个纯数据类型. 适合当没有隐式当前状态 - 只是变换的情况.
- **一个类或模块, 有清晰的方法表面** 当逻辑确实拥有持续的内部状态时.

选择最符合所问问题的形状, *不是* 最容易接线到 TUI 的. 保持纯: 无 I/O, 无终端代码, 不用 `console.log` 控制流. TUI 导入它并调用它; 没有任何东西反向流动.

这就是让原型超越其自身寿命有用的东西. 当问题被回答后, 经过验证的 reducer / machine / function set 可以提升到真实模块 - TUI shell 被删除.

### 4. 构建暴露状态的最小 TUI

构建为 **轻量 TUI** - 每个 tick, 清屏 (`console.clear()` / `print("\033[2J\033[H")` / 等效) 并重新渲染整个帧. 用户应该总是看到一个稳定的视图, 不是不断增长的滚动记录.

每个帧有两个部分, 按此顺序:

1. **当前状态**, pretty-printed 且 diff-friendly (每行一个字段, 或格式化 JSON). 使用 **bold** 标注字段名或章节标题, 使用 **dim** 标注不太重要的上下文 (时间戳, ID, 导出值). 原生 ANSI escape code 即可 - `\x1b[1m` bold, `\x1b[2m` dim, `\x1b[0m` reset. 不需要引入样式库, 除非项目中已有一个.
2. **键盘快捷键**, 列在底部: `[a] add user  [d] delete user  [t] tick clock  [q] quit`. Bold 键, dim 描述, 或反过来 - 任何看起来清晰的组合.

行为:

1. **初始化状态** - 一个单一内存对象/结构. 在启动时渲染第一帧.
2. **读取一个按键 (或一行)** 并分发到修改状态的处理器.
3. **重新渲染** 每次操作后的完整帧 - 不追加, 替换.
4. **循环直到退出.**

整个帧应适合一屏显示.

### 5. 让它一条命令可运行

向项目已有任务运行器添加脚本 (`package.json` scripts, `Makefile`, `justfile`, `pyproject.toml`). 用户应运行 `pnpm run <prototype-name>` 或等效 - 不需要记住路径.

如果宿主项目没有任务运行器, 只需把命令放在原型 README 的顶部.

### 6. 交给用户

给出运行命令. 他们会自己驱动它; 有趣的时刻是当他们说 "等等, 这不应该可能" 或 "嗯, 我以为 X 会不同" - 这些是 _想法_ 中的 bug, 这才是全部意义. 如果他们想要添加新动作, 添加它们. 原型会演化.

### 7. 捕获答案

当原型完成使命, 问题的答案才是唯一值得保留的东西. 如果用户在场, 问它教会了他们什么. 如果不在, 在原型旁留下 `NOTES.md`, 以便答案可以填入 (或由你填入, 如果你看了会话), 在原型被删除之前.

## 反模式

- **不要添加测试.** 需要测试的原型不再是原型.
- **不要连接到真实数据库.** 使用内存存储, 除非问题专门关于持久化.
- **不要泛化.** 不要 "如果我们以后想支持 X 怎么办." 原型回答一个问题.
- **不要把逻辑和 TUI 混在一起.** 如果 reducer / 状态机引用了 `console.log`, prompt, 或终端 escape code, 它不再可移植. 保持 TUI 作为纯模块的薄壳.
- **不要将 TUI shell 发布到生产.** Shell 是为从终端手动驱动而优化的. 它背后的逻辑模块才是值得保留的部分.
