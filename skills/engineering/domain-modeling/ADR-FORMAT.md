# ADR 格式

ADR 存放在 `docs/adr/` 中, 使用顺序编号: `0001-slug.md`, `0002-slug.md` 等.

惰性创建 `docs/adr/` 目录 - 仅在第一个 ADR 需要时创建.

## 模板

```md
# {决策的短标题}

{1-3 句话: 上下文是什么, 我们做了什么决定, 以及为什么.}
```

就是这样. 一个 ADR 可以只是一个段落. 价值在于记录 *做出了* 一个决策以及 *为什么* - 而不是填满各个章节.

## 可选章节

仅在它们增加真正价值时才包含. 大多数 ADR 不需要它们.

- **Status** frontmatter (`proposed | accepted | deprecated | superseded by ADR-NNNN`) - 在决策被重新审视时有用
- **Considered Options** - 仅当被拒绝的替代方案值得记住时
- **Consequences** - 仅当非显而易见的下游效应需要被指出时

## 编号

扫描 `docs/adr/` 找到最高现有编号并递增一.

## 何时提议 ADR

以下三个必须全部满足:

1. **难以逆转** - 后续改变想法的成本是显著的
2. **没有上下文会令人惊讶** - 未来读者看着代码会疑惑 "他们到底为什么要这样做?"
3. **是真正 trade-off 的结果** - 存在真实的替代方案并且你出于特定原因选择了其中一个

如果一个决策容易逆转, 跳过它 - 你以后只需要逆转. 如果它不令人惊讶, 没人会好奇为什么. 如果没有真正的替代方案, 除了 "我们做了显而易见的事", 没什么需要记录的.

### 什么符合条件

- **架构形状.** "我们使用 monorepo." "Write model 是 event-sourced 的, read model 投射到 Postgres."
- **Context 之间的集成模式.** "Ordering 和 Billing 通过领域事件通信, 不是同步 HTTP."
- **带有锁定效应的技术选择.** 数据库, 消息总线, 认证提供商, 部署目标. 不是每个库 - 只是那些需要一个季度才能替换的.
- **边界和范围决策.** "Customer 数据由 Customer context 拥有; 其他 context 仅通过 ID 引用它." 显式的 no-s 和 yes-s 一样有价值.
- **对显而易见路径的有意偏离.** "我们使用手动 SQL 而不是 ORM, 因为 X." 任何合理读者会假设相反的情况. 这些阻止下一个工程师 "修复" 一个刻意的设计.
- **代码中不可见的约束.** "我们不能使用 AWS, 因为合规要求." "响应时间必须在 200ms 以内, 因为合作方 API 合同."
- **拒绝的替代方案当拒绝原因不显而易见时.** 如果你考虑了 GraphQL 并出于细微原因选择了 REST, 记录它 - 否则六个月后有人会再次建议 GraphQL.
