# 审查

对用户提供的固定点与 `HEAD` 之间的 diff 进行双维度审查：

- **标准（Standards）**——代码是否符合此仓库记录的编码标准？
- **规格（Spec）**——代码是否忠实实现了来源 issue / PRD / spec？

两个维度都作为**并行子智能体**运行，这样它们不会相互污染上下文；随后此技能汇总它们的发现。

议题跟踪器（Issue tracker）应该已经提供给你——如果缺少 `docs/agents/issue-tracker.md`，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 固定比较点

用户说的任何内容都是固定点——提交 SHA、分支名、标签、`main`、`HEAD~5` 等。不要加入个人偏好；原样传递。如果用户没有指定，询问：“要以什么为基准审查——分支、提交，还是 `main`？”在拿到固定点之前不要继续。

一次性记录 diff 命令：`git diff <fixed-point>...HEAD`（三点，因此比较对象是合并基点 merge-base）。同时通过 `git log <fixed-point>..HEAD --oneline` 记录提交列表。

### 2. 识别规格来源

按以下顺序查找来源规格：

1. 提交消息中的 issue 引用（`#123`、`Closes #45`、GitLab `!67` 等）——通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传入的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名或功能匹配的 PRD/spec 文件。
4. 如果什么也没找到，询问用户规格在哪里。如果用户说没有规格，**规格**子智能体将跳过并报告“没有可用规格”。

### 3. 识别标准来源

仓库中任何记录代码应如何编写的内容。常见位置：

- `CLAUDE.md`、`AGENTS.md`
- `CONTRIBUTING.md`
- `CONTEXT.md`、`CONTEXT-MAP.md`、各上下文中的 `CONTEXT.md` 文件
- `docs/adr/`（架构决策也是标准）
- `.editorconfig`、`eslint.config.*`、`biome.json`、`prettier.config.*`、`tsconfig.json`（机器强制的标准——记录它们，但不要重新检查工具已经检查的内容）
- 仓库根目录或 `docs/` 下的任何 `STYLE.md`、`STANDARDS.md`、`STYLEGUIDE.md` 或类似文件

收集文件列表。**标准**子智能体会读取它们。

### 4. 并行启动两个子智能体

发送一条包含两个 `Agent` 工具调用的消息。两者都使用 `general-purpose` 子智能体。

**标准子智能体提示**——包含：

- 完整 diff 命令和提交列表。
- 第 3 步中找到的标准来源文件列表。
- 简述：“阅读标准文档。然后阅读 diff。报告——在相关时精确到文件/代码块（hunk）——diff 中每个违反已记录标准的地方。引用标准（文件 + 规则）。区分硬性违规和判断性意见。跳过工具会强制检查的内容。少于 400 字。”

**规格子智能体提示**——包含：

- diff 命令和提交列表。
- 规格的路径或获取到的内容。
- 简述：“阅读规格。然后阅读 diff。报告：(a) 规格要求但缺失或部分实现的需求；(b) diff 中未被要求的行为（范围蔓延）；(c) 看起来已实现但实现似乎错误的需求。每条发现都引用规格原文。少于 400 字。”

如果缺少规格，跳过规格子智能体，并在最终报告中说明。

### 5. 汇总

在 `## 标准（Standards）` 和 `## 规格（Spec）` 标题下展示两份报告，原样或轻微整理。**不要**合并或重新排序发现——这两个维度刻意分开，以便用户能独立查看。

以一句话总结结尾：每个维度的发现总数，以及被标记的最严重单个问题（如果有）。

## 为什么是两个维度

一项变更可能通过一个维度，却在另一个维度失败：

- 代码符合所有标准，但实现了错误的东西 → **标准通过，规格失败。**
- 代码确实做到了 issue 要求的内容，但破坏了项目约定 → **规格通过，标准失败。**

分开报告可以避免一个维度掩盖另一个维度。
