# 分流

将项目 issue tracker 上的 issue 推进一个小的 triage 角色状态机.

如果本仓库将外部 pull request 视为请求表面 (见 issue-tracker 配置), triage 也覆盖它们: **一个 PR 是附带代码的 issue** - 相同角色, 相同状态, 相同机器, 下方标注了几处 "for a PR" 的差异. 根据 tracker 配置将裸 `#42` 解析为 issue 或 PR.

triage 期间发到 issue tracker 的每条评论或 issue **必须** 以此免责声明开头:

```
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md) - 如何写持久的 agent brief
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) - `.out-of-scope/` 知识库如何工作

## 角色

两个 **类别** 角色:

- `bug` - 什么东西坏了
- `enhancement` - 新功能或改进

五个 **状态** 角色:

- `needs-triage` - 维护者需要评估
- `needs-info` - 等待报告者提供更多信息
- `ready-for-agent` - 完全明确, ready for AFK agent
- `ready-for-human` - 需要人类实现
- `wontfix` - 不会采取行动

对于 PR, 相同状态对照附带的代码读: `ready-for-agent` 意味着附有 brief, agent 应在 diff 上做下一步; `ready-for-human` 意味着它 ready for human 合并.

每个 triaged issue 应恰好携带一个类别角色和一个状态角色. 如果状态角色冲突, 标记它并在做任何其他事之前问维护者.

这些是规范角色名 - issue tracker 中实际使用的 label 字符串可能不同. 映射应该已提供给你 - 如果没有, 运行 `/setup-matt-pocock-skills`.

状态转换: 一个未标记的 issue 通常先进入 `needs-triage`; 从那里它移到 `needs-info`, `ready-for-agent`, `ready-for-human`, 或 `wontfix`. `needs-info` 在报告者回复后返回 `needs-triage`. 维护者可以随时覆盖 - 标记看起来异常的转换并在继续之前询问.

## 调用

维护者调用 `/triage` 并用自然语言描述他们想要的. 解析请求并行动. 示例:

- "显示所有需要我注意的内容"
- "Let's look at #42" (issue 或 PR)
- "将 #42 移到 ready-for-agent"
- "有哪些已准备好让 agent 接手?"

## 展示需要关注的内容

查询 issue tracker 并展示三个桶, 最旧的在前:

1. **未标记的** - 从未 triaged.
2. **`needs-triage`** - 评估进行中.
3. **`needs-info` 且报告者自上次 triage 笔记后有活动** - 需要重新评估.

当 PR 在范围内时, 将外部 PR 包含在这些桶中, 并对每行标记 `[PR]` 或 `[issue]`. 发现仅展示 *外部* PR (tracker 配置定义谁算外部) - 合作者进行中的 PR 不是 triage 工作. 此过滤器仅用于发现; 一个明确命名的 PR 无论作者身份总是被 triaged.

展示计数和每项的一行摘要. 让维护者挑选.

## 分流 一个特定 issue 或 PR

1. **收集上下文.** 读取完整的 issue 或 PR (body, 评论, labels, 作者, 日期; 对于 PR, 也读 diff). 解析任何先前的 triage 笔记, 以免重新问已解决的问题. 使用项目的领域术语表探索代码库, 尊重该区域的 ADR. 对代码库运行两项检查: (a) **冗余** - 按领域概念 (不只是请求的措辞) 搜索已有实现, 并报告你查看了哪里. 如果找到了, 它是已实现的 `wontfix` (步骤 5). (b) **先前拒绝** - 读取 `.out-of-scope/*.md` 并提出任何与此请求相似的.

2. **推荐.** 告知维护者你的类别和状态推荐及理由, 加上与请求相关的简要代码库总结 - 包括它是否已实现. 等待指示.

3. **验证主张.** 在任何 grilling 之前, 检查主张是否站得住脚. 对于 bug, 从报告者的步骤复现它. 对于 PR, 确认 diff 做了它声称的事情 - 检出, 运行相关测试或命令. 报告结果: confirmed (附代码路径), failed, 或细节不足 (一个强的 `needs-info` 信号). 一个确认的验证使 agent brief 远更有力.

4. **Grilling (如果需要).** 如果请求需要丰满, 同时运行 `/grilling` 和 `/domain-modeling` 技能 - 逐个问题将其 grilling 成形, 锐化领域术语并在决策落定时内联更新 `CONTEXT.md`/ADR.

5. **应用结果:**
 - `ready-for-agent` - 发布一个 agent brief 评论 ([AGENT-BRIEF.md](AGENT-BRIEF.md)).
 - `ready-for-human` - 同 agent brief 结构, 但注明为什么它不能被委托 (判断, 外部访问, 设计决策, 手动测试).
 - `needs-info` - 发布 triage 笔记 (下方模板).
 - `wontfix` - 关闭, 评论取决于 *原因*:
 - **已实现** - 变更已存在于代码库中. 指向它所在的位置; **不要** 写到 `.out-of-scope/` (该 KB 用于 *被拒绝* 的请求, 不是已构建的).
 - **被拒绝 (bug)** - 礼貌解释, 然后关闭.
 - **被拒绝 (enhancement)** - 写到 `.out-of-scope/`, 从评论链接到它, 然后关闭 ([OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)).
 - `needs-triage` - 应用角色. 如果有部分进展, 可选评论.

## 快速状态覆盖

如果维护者说 "move #42 to ready-for-agent", 信任他们并直接应用角色. 确认你将做什么 (角色变化, 评论, 关闭), 然后行动. 跳过 grilling. 如果在没有 grilling 会话的情况下移到 `ready-for-agent`, 问是否要写一个 agent brief.

## Needs-info 模板

```markdown
## 分流 Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

将 grilling 期间解决的所有内容捕获在 "established so far" 下, 以免工作丢失. 问题必须具体且可操作, 不是 "请提供更多信息".

## 恢复先前会话

如果 issue 或 PR 上有先前的 triage 笔记, 读取它们, 检查报告者是否回答了任何未决问题, 并在继续之前展示更新的画面. 不要重新问已解决的问题.
