# 深化

如何在给定依赖的情况下安全地深化一组浅模块. 假设使用 [SKILL.md](SKILL.md) 中的词汇 — **模块**, **接口**, **接缝**, **适配器**.

## 依赖类别

在评估深化候选对象时, 对其依赖进行分类. 类别决定了深化后的模块如何跨其接缝进行测试.

### 1. 进程内

纯计算, 内存状态, 无 I/O. 始终可深化 — 合并这些模块并直接通过新的接口进行测试. 不需要适配器.

### 2. 本地可替代

具有本地测试替身的依赖 (Postgres 的 PGLite, 内存文件系统). 如果替身存在, 则可深化. 深化后的模块在测试套件中与替身一起运行测试. 接缝是内部的; 模块的外部接口不需要端口.

### 3. 远程但自有 (端口 & 适配器)

你自己的跨网络边界服务 (微服务, 内部 API). 在接缝处定义一个 **端口** (接口). 深模块拥有逻辑; 传输作为 **适配器** 注入. 测试使用内存适配器. 生产环境使用 HTTP/gRPC/队列适配器.

推荐表述: *"在接缝处定义一个端口, 为生产环境实现一个 HTTP 适配器, 为测试实现一个内存适配器, 这样即使它跨网络部署, 逻辑仍然位于一个深模块中."*

### 4. 真正外部 (Mock)

你不控制的第三方服务 (Stripe, Twilio, 等). 深化后的模块将外部依赖作为注入端口接收; 测试提供 mock 适配器.

## 接缝纪律

* **一个适配器意味着一个假设性接缝. 两个适配器意味着一个真实接缝.** 除非至少有两个适配器是合理的 (通常是生产 + 测试), 否则不要引入端口. 单适配器接缝只是间接层.
* **内部接缝 vs 外部接缝.** 一个深模块可以有内部接缝 (对其实现私有, 由自身测试使用), 也可以有位于其接口处的外部接缝. 不要仅仅因为测试使用了内部接缝, 就把它们暴露到接口中.

## 测试策略: 替换, 不叠加

* 一旦存在位于深化后模块接口处的测试, 旧的浅模块单元测试就会变成浪费 — 删除它们.
* 在深化后模块的接口处编写新测试. **接口就是测试表面**.
* 测试应断言通过接口可观察到的结果, 而不是内部状态.
* 测试应能经受内部重构 — 它们描述行为, 而不是实现. 如果实现变化时测试也必须变化, 那它就是在越过接口进行测试.
