# UI 原型

在单一路由上生成**多个差异显著的 UI 变体**，并通过浮动底栏切换。用户在浏览器中来回切换变体，选中一个（或从每个变体里拿走一些部分），然后丢弃其余内容。

如果问题是关于逻辑/状态，而不是某个东西长什么样——那就走错分支了。使用 [LOGIC.md](LOGIC.md)。

## 什么时候适合这种形态

- “这个页面应该长什么样？”
- “在定下来之前，我想看看这个仪表盘的几个选项。”
- “给设置页面试一种不同布局。”
- 任何用户原本会花一天时间在脑中纠结三个模糊原型图的场景。

## 两种子形态——强烈优先选择子形态 A

UI 原型在**贴着应用其余部分**时更容易评判——真实页头、真实侧边栏、真实数据、真实密度。孤立的一次性路由是个真空：每个变体单独看都还行。只要有合理的现有页面可以承载变体，默认使用子形态 A。只有当原型确实没有邻近归属时，才选择子形态 B。

### 子形态 A——调整现有页面（首选）

路由已经存在。变体在**同一路由**上渲染，由 `?variant=` URL 查询参数控制。现有的数据获取、params 和 auth 都保留——只替换渲染内容。这是默认选择；除非有明确理由，否则就选它。

如果原型验证的是还没有页面、但*自然会位于某个页面内部*的东西（仪表盘的新分区（section）、设置页面的新卡片（card）、现有流程的新步骤（step））——这仍然是子形态 A。把变体挂载到宿主页面内部。

### 子形态 B——新页面（最后手段）

只有当被原型验证的东西确实没有现有页面可容纳时才使用——例如一个全新的顶层表面，或一个无法合理嵌入任何地方的流程。

按照项目既有路由约定创建一个**一次性路由**——不要发明新的顶层结构。命名要明显表明它是原型（例如在路径或文件名里包含 `prototype`）。同样使用 `?variant=` 模式。

在决定使用子形态 B 前，先做一次合理性检查：真的没有现有页面可以把它嵌进去吗？空路由会隐藏有内容的页面本会暴露的设计问题。

两种子形态中的浮动底栏完全相同。

## 流程

### 1. 陈述问题并选择 N

默认使用 **3 个变体**。超过 5 个就不再是差异显著，而会变成噪声——最多 5 个。

在原型位置或文件顶部注释中，用一行写下计划：

> “设置页面的三个变体，在现有 `/settings` 路由上通过 `?variant=` 切换。”

无论用户是否在场可以反驳，这都有效。

### 2. 生成差异显著的变体

起草每个变体。每个都要遵守：

- 页面目的和它能访问的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、普通 CSS（plain CSS），随项目而定）。
- 清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须在**结构上不同**——不同布局、不同信息层级、不同主要操作入口，而不只是不同颜色。三个轻微调整过的卡片网格（card grid）不是 UI 原型，而是壁纸。如果两个草稿太相似，就明确给出“不要使用卡片网格（card grid）”的指导并重做其中一个。

### 3. 将它们接在一起

在路由上创建一个单一切换器（switcher）组件：

```tsx
// 伪代码——按项目框架调整
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：保留切换器（switcher）上方所有现有数据获取；只有渲染子树会随变体改变。

对于子形态 B（新页面）：`/prototype/<name>` 下的一次性路由挂载同一个切换器（switcher）。

### 4. 构建浮动切换器（switcher）

一个固定定位在屏幕底部居中的小栏，包含三部分：

- **左箭头**——循环到上一个变体（到头后回绕）。
- **变体标签**——显示当前变体键（key）；如果变体导出了名称，也显示名称。例如 `B — 侧边栏布局`。
- **右箭头**——循环到下一个变体（到头后回绕）。

行为：

- 点击箭头会更新 URL 查询参数（使用框架的路由器（router）——Next 中用 `router.replace`，React Router 中用 `navigate` 等），让变体可分享且刷新后稳定。
- 键盘：`←` 和 `→` 方向键也会循环。不要在 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时拦截方向键。
- 视觉上与页面区分开（例如高对比度胶囊、轻微阴影），让人明显看出它不是正在评估的设计的一部分。
- 在生产构建中隐藏——通过 `process.env.NODE_ENV !== 'production'` 或等价检查做门控（gate），这样即使原型意外合并，也不会把底栏发给用户。

把 switcher 放进一个共享组件，让两种子形态都能复用。把它放在项目存放共享 UI 的位置。

### 5. 交付给用户

暴露 URL（以及 `?variant=` 键）。用户会在方便时自己切换查看。最有价值的反馈通常是**“我想要 B 的页头配 C 的侧边栏”**——那才是他们真正想要的设计。

### 6. 捕获答案并清理

一旦某个变体胜出，写下是哪一个以及原因（提交消息（commit message）、ADR、issue；如果 AFK 运行且用户还没回应，就在原型旁边写 `NOTES.md`）。然后：

- **子形态 A**——删除落选变体和切换器（switcher）；把胜出方案折叠进现有页面。
- **子形态 B**——把胜出变体提升为真实路由，删除一次性路由和切换器（switcher）。

不要把变体组件或切换器（switcher）留在周围。它们很快会腐烂，并让下一位读者困惑。

## 反模式

- **只在颜色或文案上不同的变体。** 那是微调，不是原型。真正的变体会在结构上产生分歧。
- **在变体之间共享过多代码。** 共享 `<Header>` 没问题；共享 `<Layout>` 会破坏目的。每个变体都应该可以自由丢弃布局。
- **把变体接到真实变更操作（mutation）上。** 只读原型是可以的。如果某个变体需要修改数据，把它指向存根（stub）——问题是“这应该长什么样”，不是“后端是否可用”。
- **直接把原型提升到生产。** 变体代码是在原型约束下编写的（无测试、最小错误处理）。折叠进真实代码时要正确重写。
